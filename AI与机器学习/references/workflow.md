# 工作流规范

## 001 摄入流程（Ingest）

1. 原始资料先放入 `raw/` 或 `raw/staging/`。
2. 不直接把原文搬进 `wiki/`。
3. 先写来源摘要或来源地图。
4. 再编译为正式知识页。
5. 更新 `wiki/index.md`。
6. 更新 `wiki/log.md`。

## 002 查询流程（Query）

1. 先读 `wiki/index.md`。
2. 再读 `wiki/00-当前主路径.md`。
3. 涉及当前决策时，先读 `wiki/outputs/当前决策总账.md`。
4. 再进入对应主题页。
5. 有复用价值的回答，沉淀到 `wiki/outputs/`。

## 003 决策固化流程（Decision）

1. 待裁决事项先进入 outputs 或对应主题页。
2. 用户确认后，写入 `wiki/outputs/当前决策总账.md`。
3. 若是阶段性裁决，另建正式裁决页。
4. 决策必须包含：结论、边界、适用范围、来源路径。
5. 更新 `wiki/index.md` 和 `wiki/log.md`。

## 004 来源地图维护流程（Source Map）

1. 大批量资料必须建立来源地图。
2. 来源地图至少包含：文件路径、主题分类、是否已编译、编译到哪个页面、是否需要人工复核。
3. 运维、配置、服务器、连接相关资料默认需要人工复核。
4. 来源地图不复制敏感内容，只记录治理信息。

## 005 当前主路径更新流程（Main Path）

以下事项变化时，必须更新 `wiki/00-当前主路径.md`：

- 当前阶段目标变化；
- 首脑 / Runtime / Codex 口径变化；
- 是否重装服务器变化；
- 是否进入数仓样板链路变化；
- 是否允许生产动作变化。

## 006 状态管理

页面状态使用：

```text
stable：已确认，可作为执行依据
draft：草案，不能直接执行
legacy：旧版保留，仅供追溯
deprecated：已废弃，不再作为依据
```

`stable` 页面必须有来源路径。  
`draft` 页面必须写待裁决事项。  
`legacy/deprecated` 页面不得出现在当前主路径。

## 007 健康检查（Health）

定期执行：

```bash
python tools/health.py --json
```

结构检查项：

- 必要文件是否存在；
- 页面是否为空；
- frontmatter 是否存在；
- index 链接是否断裂；
- wiki 页面是否进入 index；
- 是否出现明显敏感字段；
- raw 引用是否存在。

治理检查项：

- `wiki/00-当前主路径.md` 是否存在；
- `wiki/outputs/当前决策总账.md` 是否存在；
- `wiki/index.md` 是否包含当前主路径；
- `wiki/index.md` 顶部统计是否过时；
- 新增阶段是否写入 `wiki/log.md`。

## 008 禁止事项

- 不把 raw 当正式知识页；
- 不把临时讨论当 stable 决策；
- 不把敏感配置写入 wiki 正文；
- 不删除旧页来制造“干净”；
- 不绕过 index/log 直接沉淀；
- 不让 Codex / Runtime / 首脑口径混用。
