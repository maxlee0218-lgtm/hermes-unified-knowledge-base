# 运行治理总手册

## 1. 结论

Runtime 是运行治理体系，不是首脑。Runtime 的职责是看状态、管生命周期、识别堵点、输出运行总账；首脑负责统筹判断、知识沉淀和结果汇报。

## 2. 适用范围

适用于任务状态巡检、Review 堆积处理、Heartbeat 判断、GitHub 任务总线检查、done/archive 收口和运行异常分诊。

## 3. 已确认事实

- 历史资料显示 Runtime 曾承担过治理、巡检、Heartbeat、Review loop、任务闭环等职责；
- Multica 更适合作为 Issue / Agent Runtime 执行池，不适合作为完整总控台；
- GitHub 更适合作为指令、审计、知识沉淀层；
- 运行状态判断需要统一总账，不能只看本地目录。

## 4. 核心规则

1. Runtime 只做运行治理，不做业务产物；
2. Multica 是执行池，不是知识库；
3. GitHub 是审计层，不是真实运行态唯一来源；
4. 任务状态必须区分 pending、active、review、done、blocked、failed、archived；
5. 任何生产动作必须人工确认；
6. 运行报告必须落库到 wiki 或 20-resources。

## 5. 操作流程

1. 读取 GitHub index/log 和 runtime 状态文件；
2. 读取 Multica Issue 状态；
3. 对比 pending/done/failed 与实际 active/review；
4. 标出 review 堆积、active 超时、blocked、failed、重复任务；
5. 输出运行治理报告；
6. 将稳定规则沉淀进 wiki/runtime。

## 6. 风险与限制

- 单看 GitHub 目录会产生“幽灵任务”；
- 单看 Multica 会缺少知识沉淀；
- Runtime 与首脑混用会导致职责混乱；
- 自动关闭或修改状态风险高，应保留人工确认。

## 7. 待裁决事项

- Runtime Flow Guard 是否作为新主链路必备组件；
- Multica 是否继续作为执行池保留；
- 运行总账文件的最终位置和刷新频率。

## 8. 后续动作

- 建立标准运行状态页；
- 建立 Review SLA；
- 建立 done/archive 归档机制；
- 每次运行治理输出必须更新 index/log。
