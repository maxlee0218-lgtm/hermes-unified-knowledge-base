       ▼

## 二、任务流转图

### 2.1 旧流转（已废弃）

```
[ChatGPT 写任务] → [推送到 GitHub] → [PocketClaw 扫描]
                                              ↓
                                    [task_inbox_check.sh]
                                              ↓
                                    [git pull 同步]
                                              ↓
                                    [识别 inbox 新任务]
                                              ↓
                                    [边界校验]
                                              ↓
                                    [创建 Multica Issue]
                                              ↓
                                    [移动到 active]
                                              ↓
                                    [Agent 执行]
                                              ↓
                                    [REVIEW / DONE]
                                              ↓
                                    [LLM Wiki / GitHub 沉淀]
```

### 2.2 新流转（当前）

```
[用户告知任务] ───────────────────────────────┐
                                              ↓
[首脑直接创建 Multica Issue] ───────┐         │
                                    ↓         │
[指派给 研究与架构官]              │         │
                                    ↓         │
[Agent 自动执行] ───────────────────┘         │
                                              ↓
[输出报告到 Multica 评论]                     │
                                              ↓
[首脑检查报告] ←──────────────────────────────┘
                                              ↓
[验收通过 → 转 done]
                                              ↓
[归档到本地 done 目录]
                                              ↓
[更新 LLM Wiki]
                                              ↓
[GitHub 同步（可选）]
```

## 四、角色职责

### 4.1 用户
- 告知任务目标
- 确认验收结果

### 4.2 首脑（150服务器）
- 创建 Multica Issue
- 指派 Agent
- 验收报告
- 归档任务

### 4.3 研究与架构官（Agent）
- 执行任务
- 输出报告
- 评论更新

---

## 五、关键变更点

| 变更 | 说明 |
|------|------|
| 入口变更 | GitHub → 用户直接告知 |
| 创建变更 | 自动脚本 → 首脑手动创建 |
| 指派变更 | 按规则自动 → 统一指派研究与架构官 |
| 同步变更 | 强制同步 → 可选同步 |

## 六、优势对比

| 维度 | 旧架构 | 新架构 |
|------|--------|--------|
| 响应速度 | 慢（需等待同步） | 快（立即创建） |
| 复杂度 | 高（多环节） | 低（直接） |
| 可靠性 | 中（依赖GitHub） | 高（直接控制） |
| 灵活性 | 低（固定流程） | 高（首脑判断） |
| 可追溯 | 高（Git历史） | 中（Multica记录） |
