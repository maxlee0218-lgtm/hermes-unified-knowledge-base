# bigdata-etl-cr

## 功能描述

对数仓 ETL 代码进行 Code Review，分析代码逻辑的合理性，有效发现潜在的逻辑问题、性能问题和数据质量问题。

## 支持的代码类型

- Hive SQL
- Spark SQL
- Flink SQL
- DataWorks / MaxCompute SQL
- 其他大数据 SQL 方言

## 使用方法

用户提供 ETL 代码后，将按照以下维度进行审查：

## CR 检查清单

### 1. 数据正确性

#### 1.1 JOIN 逻辑
- [ ] JOIN 条件是否正确，是否遗漏关联字段
- [ ] JOIN 类型是否合理（LEFT/INNER/RIGHT）
- [ ] 是否存在一对多导致数据膨胀
- [ ] 是否存在笛卡尔积
- [ ] 关联键数据类型是否一致

#### 1.2 过滤条件
- [ ] WHERE 条件是否完整
- [ ] 是否正确处理 NULL 值
- [ ] 时间过滤条件是否正确
- [ ] 业务状态过滤是否遗漏
- [ ] 分区过滤是否生效

#### 1.3 聚合逻辑
- [ ] GROUP BY 字段是否完整
- [ ] 聚合函数使用是否正确（SUM/COUNT/DISTINCT）
- [ ] 是否存在数据倾斜风险
- [ ] 去重逻辑是否正确

#### 1.4 计算逻辑
- [ ] 算术运算是否考虑 NULL 和除零
- [ ] 金额计算精度是否正确（分/元转换）
- [ ] 比例/百分比计算逻辑
- [ ] 时间差计算是否正确

### 2. 数据质量

#### 2.1 空值处理
- [ ] 关键字段是否有 NULL 检查
- [ ] COALESCE/NVL 使用是否正确
- [ ] 空字符串与 NULL 是否区分

#### 2.2 数据去重
- [ ] 是否存在重复数据风险
- [ ] 去重逻辑是否正确（ROW_NUMBER/GROUP BY）
- [ ] 主键是否唯一

#### 2.3 数据类型
- [ ] 类型转换是否正确（CAST）
- [ ] 字符串与数值比较是否合理
- [ ] 时间类型格式是否统一

### 3. 性能优化

#### 3.1 分区剪枝
- [ ] 是否正确使用分区字段过滤
- [ ] 动态分区是否合理
- [ ] 分区粒度是否合适

#### 3.2 数据倾斜
- [ ] 是否存在热点 Key
- [ ] 大表 JOIN 是否优化
- [ ] GROUP BY 是否需要分散

#### 3.3 资源使用
- [ ] 是否有不必要的大表扫描
- [ ] 中间结果是否合理
- [ ] 是否可以增量处理

### 4. 代码规范

#### 4.1 可读性
- [ ] 字段命名是否清晰
- [ ] 是否有必要的注释
- [ ] SQL 格式是否规范
- [ ] 复杂逻辑是否有说明

#### 4.2 可维护性
- [ ] 是否有硬编码
- [ ] 魔法数字是否有注释说明
- [ ] 是否方便调试和排查

### 5. 常见逻辑陷阱

#### 5.1 时间相关
- [ ] 跨天数据处理
- [ ] 时区问题
- [ ] 月末/年末边界
- [ ] 闰年处理

#### 5.2 状态流转
- [ ] 状态机是否完整
- [ ] 是否遗漏中间状态
- [ ] 终态处理是否正确

#### 5.3 业务规则
- [ ] 业务规则理解是否正确
- [ ] 特殊场景是否覆盖
- [ ] 逆向推导是否成立

## CR 示例

### 输入代码

```sql
SELECT 
    a.order_id,
    a.user_id,
    b.user_name,
    SUM(a.amount) as total_amount
FROM orders a
LEFT JOIN users b ON a.user_id = b.user_id
WHERE a.order_status = 2
GROUP BY a.order_id
```

## 常见问题模式库

### 问题 1：一对多 JOIN 导致数据膨胀
```sql
-- 问题代码
SELECT a.*, b.*
FROM table_a a
LEFT JOIN table_b b ON a.id = b.a_id

-- 问题：如果 table_b 中一个 a_id 对应多条记录，会导致结果行数增加
-- 建议：确认业务需求，可能需要聚合或添加额外条件
```

### 问题 2：NULL 值陷阱
```sql
-- 问题代码
WHERE status != 'deleted'

-- 问题：status 为 NULL 的记录会被过滤掉
-- 建议：WHERE (status IS NULL OR status != 'deleted')
```

### 问题 3：除零风险
```sql
-- 问题代码
SELECT amount / quantity

-- 问题：quantity 为 0 时会报错或返回 NULL
-- 建议：CASE WHEN quantity = 0 THEN 0 ELSE amount / quantity END
```

### 问题 4：时间比较类型不一致
```sql
-- 问题代码
WHERE dt = 20240101

-- 问题：dt 是字符串类型，数值比较可能出问题
-- 建议：WHERE dt = '20240101'
```

### 问题 5：COUNT 误用
```sql
-- 问题代码
SELECT COUNT(status)

-- 问题：COUNT(字段) 不统计 NULL，可能不是预期行为
-- 建议：确认是否需要 COUNT(*) 或 COUNT(DISTINCT status)
```

## 注意事项

- CR 结果仅供参考，需结合业务场景判断
- 重点关注数据正确性和逻辑问题
- 对于复杂业务逻辑，建议与业务方确认
- 发现问题时应给出具体的修复建议
- 对于可疑但不确定的问题，应标注为“疑似问题”并建议确认

## 使用技巧

1. **提供上下文**：提供表结构、数据样例可提高 CR 准确性
2. **说明业务背景**：业务规则说明有助于判断逻辑正确性
3. **标注关注重点**：可指定重点关注的检查维度
