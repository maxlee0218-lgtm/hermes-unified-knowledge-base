## 建议测试路径

### 1. 看服务是否活着

打开：

- `http://127.0.0.1:8020/health`
- `http://127.0.0.1:8020/queue`
- `http://127.0.0.1:8020/certificates`

### 2. 看队列是否在推进

优先检查 `query_queue`：

```sql
select status, count(*) as cnt
from query_queue
group by status
order by status;
```

```sql
select id, name, id_card, status, result_count, new_cert_count, duration_seconds, error_message
from query_queue
order by id;
```

### 3. 看证书是否持续落库

```sql
select count(*) as cert_count from certificates;
```

```sql
select e.name, c.certificate_type, c.cert_item, c.cert_number, c.issue_authority, c.valid_until, c.status
from certificates c
left join employees e on e.id = c.employee_id
order by c.id desc
limit 50;
```

### 4. 看查询日志是否同步写入

```sql
select count(*) as log_count from query_logs;
```

```sql
select id, name, id_card, query_type, query_source, result_count, query_status, message, queried_at
from query_logs
order by id desc
limit 50;
```

### 5. 页面联动检查

至少验证这些页面：

- `/queue`
- `/certificates`
- `/employees`
- `/search`
- `/expiring`

- 队列统计数字有变化

- 搜索姓名或身份证能搜到新结果
- 即将到期页不会报错

## 已知现象

- `特种设备作业人员证` 链路整体更稳
- `特种作业操作证` 链路能出结果，但波动仍可能更大
- 部分人员会查出多条证书
- 部分记录 `valid_until` 可能为空，但仍返回有效结果
- 本机无法连远端 MariaDB，当前一律落本地 SQLite

## 队列节奏参数

当前 `.env` 中的关键配置：

- `QUERY_DELAY_MIN=2`
- `QUERY_DELAY_MAX=5`
- `QUEUE_DELAY_MIN=6`
- `QUEUE_DELAY_MAX=14`
- `QUEUE_BATCH_PAUSE_MIN=12`
- `QUEUE_BATCH_PAUSE_MAX=28`
- `QUEUE_SHUFFLE_WINDOW=6`

含义：

- 单次查询内部有基础延迟
- 队列条间会再随机停顿
- 每处理 3 条会额外休息一段
- 同优先级任务会在小窗口内打散顺序

## 如果要继续跑批

先看还有没有待处理：

```sql
select count(*) from query_queue where status = '待处理';
```

如果要继续处理，优先走系统现有接口，而不是另起一套逻辑：

- 页面按钮：`/queue`
- API：`POST /api/queue/trigger`

## 如果要判定“卡住”

可以用下面的标准：

- `10` 分钟内 `query_queue` 的完成数完全不变
- `query_logs` 没有新增
- `certificates` 没有新增或更新
- 进程仍在，但队列长期停在同一条
