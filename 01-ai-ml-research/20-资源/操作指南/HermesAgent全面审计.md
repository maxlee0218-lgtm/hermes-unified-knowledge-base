## 二、发现的缺陷与风险

### 🔴 高优先级

#### 1. 版本严重落后（核心风险）

- **当前**：v0.10.0 (commit c681687, 2026-04-22)
- **最新**：v2026.4.30（日历版本）或 v0.12.0（语义版本）
- **差距**：约 2 周开发量，数百个文件变更
- **风险**：
  - 已知 bug 未修复（如 gateway 稳定性、内存泄漏）
  - 新功能不可用（如改进的 memory provider、kanban 增强）
  - 安全补丁缺失
- **阻塞原因**：服务器到 GitHub 下载速度 ~30KB/s，18MB tarball 无法在 600s 工具超时内完成
- **缓解**：已尝试 5 次下载均失败，需用户本地下载后 scp 上传

#### 2. 磁盘空间 69%（监控阈值）

- **当前**：26G/40G 已用
- **风险**：日志、缓存、备份持续增长，可能在未来 1-2 个月内触发报警
- **缓解**：已配置每周清理 timer，保留策略 7-30 天不等
- **建议**：当达到 80% 时，清理旧版本备份目录或扩展磁盘

#### 3. Git 工作树大量未跟踪文件（+ 大量修改）

- **修改文件**：500+（`M` 状态）
- **未跟踪文件**：200+（`??` 状态）
- **风险**：
  - 无法通过 `git status` 快速判断真实变更
  - 升级时可能冲突
  - 本地修改与上游更新难以合并
- **根因**：v0.10.0 代码库与当前运行环境存在大量本地变更（可能是安装时生成或手动修改）
- **建议**：升级前必须理清这些变更，或采用「全新安装」策略而非 git pull

### 🟡 中优先级

#### 4. PolarDB 隧道双进程冗余

- **现象**：`sshpass` 和 `ssh` 各一个进程维护同一隧道（15018）
- **风险**：资源浪费，端口冲突隐患
- **建议**：检查 systemd service 配置，避免重复启动

#### 5. 日志中的 Shutdown Diagnostic

- **现象**：gateway.log 中出现 "Shutdown diagnostic — other hermes processes running"
- **含义**：gateway 重启时检测到其他 Hermes 进程，可能是之前 kill 不彻底
- **风险**：session 状态可能不一致
- **建议**：下次重启前确认 `pkill -9` 已完全清理

#### 6. 配置冗余：api_key 重复存储

- **现象**：`custom_providers` 和 `providers.kimicode` 使用同一 api_key
- **风险**：key 泄露时影响面扩大
- **建议**：111 专用 key 已配置，但 `providers.kimicode` 段可删除（如果只用 custom）

### 🟢 低优先级

#### 7. 技能目录膨胀

- **现象**：`~/.hermes/skills/` 有大量可选技能，部分可能从未使用
- **影响**：启动时扫描耗时，prompt 中 skill 引用可能冗余
- **建议**：定期清理未使用技能（通过 `hermes skills` 查看活跃度）

#### 8. 未使用平台配置残留

- **现象**：config.yaml 中可能包含未启用的平台（如 telegram、discord）的占位配置
- **影响**：config 文件体积增大，解析耗时
- **建议**：删除未使用平台的配置段

## 三、可提升的地方

### 1. 架构层面

| 领域 | 当前 | 可提升 |
|------|------|--------|
| 记忆系统 | 纯文本文件，2200/1375 chars 限制 | SQLite + FTS5 检索，按需召回 |
| 版本更新 | 手动下载 tarball，网络阻塞 | 配置 HTTP 代理或本地镜像 |
| 监控告警 | 无自动磁盘/内存监控 | 接入现有钉钉告警通道 |
| 备份策略 | 手动 cp 备份 | 自动快照 + 异地同步（Git） |

### 2. 运维层面

| 领域 | 当前 | 可提升 |
|------|------|--------|
| 进程管理 | nohup + 手动 pkill | systemd service 托管 |
| 日志轮转 | 无，gateway.log 持续增长 | logrotate 或按大小切割 |
| 配置管理 | 单文件 config.yaml | 分环境配置（dev/prod） |
| Key 管理 | 明文存储在 config.yaml | 环境变量注入 + 权限隔离 |

### 3. 功能层面

| 领域 | 当前 | 可提升 |
|------|------|--------|
| 多实例协调 | 111 + 150 各自独立 | 统一控制面板或主从架构 |
| 会话恢复 | 依赖 session_search | 持久化会话状态到 DB |
| 技能更新 | 手动 git pull | 自动检测 + 增量更新 |
| 模型切换 | 改 config + 重启 gateway | 热切换 API（如 /model 命令生效） |
