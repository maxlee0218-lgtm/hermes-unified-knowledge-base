# 记忆系统优化方案

> 分析当前 Hermes Agent 内置记忆系统的局限，提出可落地的改进路径。

---

## 一、当前系统诊断

### 1.1 现状

| 维度 | 当前实现 |
|------|---------|
| 存储 | 纯文本文件 `MEMORY.md` + `USER.md` |
| 分隔 | `§` (section sign) |
| 注入 | 启动时冻结快照，会话中写入不更新 prompt |
| 容量 | MEMORY 2200 chars, USER 1375 chars |
| 操作 | add/replace/remove（基于子串匹配） |
| 位置 | `~/.hermes/memories/` |

### 1.2 已发现问题

**容量压力**
- MEMORY.md: 4732 bytes / 2200 limit = **215% 超载**（实际 18 条，但字符数已超）
- USER.md: 1427 bytes / 1375 limit = **104% 超载**
- 系统仍在工作是因为 `load_from_disk()` 加载时不强制截断，但新增写入会失败

**信息过期**
- Phoenix V5.1.1 已关闭（2026-05-06），但 4 条相关记忆仍占用空间
- 150 服务器已卸载 Phoenix，但额度策略条目仍在
- 监控大屏问题可能已修复，但负面描述仍影响决策

**结构缺陷**
- 无标签/分类：无法区分「基础设施」「项目状态」「用户偏好」
- 无优先级：重要约束（如"不传输私钥"）和次要信息（如"D盘监控任务"）同等对待
- 无关联：条目孤立，无法形成知识网络
- 无检索：全量注入，不能按需召回

---

## 二、优化方向

### 2.1 短期：存量治理（立即可做）

**目标**：在现有架构内，释放容量、提升信息质量

#### Step 1: 清理过期条目

```bash
# 删除 Phoenix 已关闭的冗余条目
# 合并 150 服务器两条为一条精简版
# 更新监控大屏状态（如果已修复）
```

**预期释放**：~800 chars（约 36% 容量）

#### Step 2: 压缩长条目

将大段文字改为结构化短句：

```
# 改造前（证照系统，~200 chars）
证照系统V1阶段（AI表格录入）已完成小范围试点，验证提效948条录入节省23.7小时...

# 改造后（~80 chars）
证照系统: V1试点完成(948条/23.7h提效), V2全公司推广中, 官网抓取验证中
```

#### Step 3: 拆分 USER.md

当前 USER.md 混合了：
- 用户身份（全栈工程师、Python后端）
- 对话偏好（简洁、结果导向）
- 核心关注（安全、连接、数仓、证书）
- 禁忌（不翻译、不筛选确认、不明文传私钥）
- 格式约束（直接执行/深度交互）

建议拆分为：
- `USER.md` — 身份 + 偏好（核心，高频引用）
- `CONSTRAINTS.md` — 禁忌 + 格式约束（次核心）
- `PROFILE.md` — 关注领域 + 技能栈（低频，按需加载）

### 2.2 中期：结构增强（需改代码）

**目标**：在不换存储介质的前提下，增加元数据层

#### 方案 A: 前置标签

在条目内增加轻量标记：

```markdown
§
[infra:stable] 跳板机: Ubuntu 111.229.153.11, 本地文件系统即跳板机
[project:active] 证照系统: V2推广中, 阿里云 8.163.49.184
[security:critical] 禁忌: 不明文传输私钥, 不接受翻译
```

**好处**：
- 无需改存储格式
- 模型可通过标签快速识别类别
- 便于未来脚本化过滤

**代价**：
- 标签本身占字符
- 依赖模型理解标签语义

#### 方案 B:  companion JSON 索引

每条记忆配一个 `memories/index.json`：

```json
{
  "entries": [
    {
      "id": "phoenix-v51",
      "file": "MEMORY.md",
      "offset": 0,
      "length": 120,
      "tags": ["phoenix", "infra"],
      "status": "archived",
      "priority": "low",
      "created": "2026-04-28",
      "updated": "2026-05-06"
    }
  ]
}
```

**好处**：
- 不占用 prompt 空间
- 支持按标签/状态/优先级过滤
- 可扩展为自动归档机制

**代价**：
- 需要改 `memory_tool.py` 维护索引一致性
- 增加文件 IO

### 2.3 长期：架构升级（需评估）

#### 方案 C: SQLite 存储 + FTS5 检索

将 `MEMORY.md` + `USER.md` 迁移到 `~/.hermes/memories.db`：

```sql
CREATE TABLE memories (
    id TEXT PRIMARY KEY,
    target TEXT CHECK(target IN ('memory', 'user')),
    content TEXT,
    tags TEXT, -- JSON array
    priority INTEGER DEFAULT 5, -- 1-10
    status TEXT DEFAULT 'active', -- active | archived | draft
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    access_count INTEGER DEFAULT 0,
    last_accessed TIMESTAMP
);

CREATE VIRTUAL TABLE memories_fts USING fts5(content, tags);
```

**好处**：
- 容量不再受字符限制（受磁盘限制）
- 支持全文检索：按需召回，不全量注入
- 支持自动归档：status='archived' 的条目不注入 prompt
- 支持访问统计：自动识别高频/低频记忆
- 与现有 `hermes_state.py`（SQLite session store）技术栈一致

**代价**：
- 需要重写 `memory_tool.py` 和 `MemoryStore`
- 需要迁移现有 `.md` 文件
- 需要处理并发写入（但现有 flock 机制可复用）

#### 方案 D: 向量检索 + 语义召回

引入轻量嵌入模型（如 `sentence-transformers/all-MiniLM-L6-v2`），将记忆向量化：

```python
# 伪代码
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('all-MiniLM-L6-v2')

# 写入时生成向量
embedding = model.encode(content)
store_in_faiss_or_sqlite(embedding, content, metadata)

# 查询时语义召回
query_embedding = model.encode(user_message)
relevant = search_nearest(query_embedding, k=5)
```

**好处**：
- 语义召回：用户说"修一下大屏"能召回「监控大屏」相关记忆，即使关键词不匹配
- 动态容量：只注入最相关的 5 条，不受总容量限制

**代价**：
- 增加依赖（~400MB 模型）
- 嵌入计算有延迟（但可后台预计算）
- 需要改 `prefetch()` 逻辑

## 三、推荐路径

### 立即执行（今天）

1. **清理过期记忆**：删除 Phoenix 相关 4 条，合并 150 服务器 2 条
2. **压缩长条目**：证照系统、监控大屏等长文本精简
3. **验证容量**：确保清理后 MEMORY < 2000 chars, USER < 1300 chars

### 本周内

4. **增加标签**：在新增/修改条目时，前置 `[category:status]` 标记
5. **建立归档约定**：已关闭/完成的项目，条目首行加 `[archived]`

### 本月内（如需）

6. **评估 SQLite 方案**：如果标签方案仍无法解决容量问题，实施 SQLite + FTS5
7. **保留向量方案**：作为未来选项，等 Hermes 官方支持或社区插件
