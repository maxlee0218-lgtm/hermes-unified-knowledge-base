# 三类Agent角色摸底、流程设计与协同规范

## 背景
当前系统主要有三个执行/研究角色：
- 数据专家
- 数仓管家
- 深度研究智能体

用户要求：在继续推进数仓重构、Windows环境治理、四机资产盘点之前，必须先把这三个角色摸清楚，判断它们是否合适，并给它们设计流程、设立角色、制定规范。如果当前角色不好，要先从工作模式和协同方式开始改造，增强它们的工作能力。

## 核心目标
回答以下问题：

1. 当前三个角色到底分别擅长什么；
2. 当前三个角色分别不适合做什么；
3. 当前任务分配是否合理；
4. 哪些任务发错了角色；
5. 哪些任务因为角色边界不清导致卡住；
6. 三个角色之间应该如何交接；
7. 主脑应该如何验收它们；
8. 是否需要新增第四类角色；
9. 是否需要合并或重命名角色；
10. 如何让后续任务更稳定闭环。

## 必须摸底范围

### 001 数据专家
重点分析：
- 已执行哪些任务；
- 成功案例；
- 失败案例；
- 是否适合 Windows 本地执行；
- 是否适合 SQL / 数据质量 / 沙箱验证；
- 是否适合读 C/D 盘；
- 是否容易误操作；
- 是否需要更严格的只读边界；
- 标准输入和输出应是什么。

### 002 数仓管家
重点分析：
- 是否适合数仓链路、Dolphin、DataX、ODS/DWD/ADS；
- 是否适合调度排查；
- 是否适合重构前基线；
- 是否存在 timeout 或执行不稳定；
- 是否应该只做只读分析，不做执行；
- 与数据专家如何分工。

### 003 深度研究智能体
重点分析：
- 是否适合方法论、规划、尽调、规范、架构；
- 是否适合长期知识沉淀；
- 是否适合异常分诊；
- 是否不应该接执行型任务；
- 如何和主脑区分职责；
- 如何避免变成“空泛总结”。

## 必须输出的规范

### A. 角色职责表
字段：
- 角色名
- 定位
- 适合任务
- 不适合任务
- 允许动作
- 禁止动作
- 标准输入
- 标准输出
- 验收标准
- 常见失败模式

### B. 任务分配矩阵
必须覆盖：
- DATA
- SANDBOX
- SQL
- WAREHOUSE
- PLAN
- REVIEW
- KNOWLEDGE
- INVENTORY
- MIGRATION_PLAN
- RUNTIME_MONITOR
- INCIDENT_TRIAGE

### C. 协同流程
至少覆盖：

1. 新任务进入：ChatGPT → GitHub → PocketClaw → Multica；
2. 执行任务：Agent → 报告 → 主脑；
3. 验收任务：主脑 → REVIEW → DONE；
4. 知识沉淀：Wiki/GitHub；
5. 异常任务：blocked/failed/review_stuck → 分诊 → 用户决策。

### D. 交接规范
例如：
- 深度研究智能体输出方案后，如何交给数据专家；
- 数仓管家输出链路后，如何交给数据专家验证；
- 数据专家输出Windows摸排后，如何回写给数仓管家和深度研究智能体。

### E. 异常升级规则
必须明确：
- 什么情况直接标 blocked；
- 什么情况进入 review；
- 什么情况需要用户确认；
- 什么情况禁止继续重试；
- 什么情况只保留观察。

## 关键判断
必须判断：

001 当前三个角色是否足够？
002 是否需要新增“运维/资产管家”角色？
003 是否需要新增“验收官/审计官”角色？
004 是否需要把主脑从执行任务中彻底剥离？
005 是否需要建立“角色能力档案”？
