# 任务生命周期治理手册

## 1. 结论

任务治理不能只看“创建”和“完成”，必须覆盖完整生命周期：待处理、执行中、待审核、阻塞、失败、完成、归档。历史问题的根因不是没有任务，而是任务状态没有闭环。

## 2. 适用范围

适用于 GitHub Issue、Multica Issue、runtime-commands、review loop、done/archive 归档和 blocked/failed 分诊。

## 3. 已确认事实

- 历史报告中多次出现 review 堆积、done 未归档、blocked 未分诊、active 超时；
- review loop 已验证可用，但不能自动越权处理；
- done 不等于最终闭环，必须有报告、证据、索引和归档；
- cancelled 或测试类任务需要最小归档，避免长期污染 active/review。

## 4. 核心规则

1. 每个任务必须有目标、边界、验收标准；
2. in_review 任务必须按 SLA 分诊；
3. blocked 必须写明阻塞原因和下一步；
4. done 必须有可追溯证据；
5. archived 只接收已闭环任务；
6. 重复任务必须合并或标记，不得并行消耗资源。

## 5. 操作流程

1. 拉取任务总表；
2. 分类：active / review / done / blocked / failed / cancelled；
3. 对 review 任务判断 ready、missing evidence、blocked、duplicate；
4. 对 done 任务检查报告和索引；
5. 对 blocked 任务输出人工决策项；
6. 输出生命周期治理报告；
7. 将可复用规则沉淀到 wiki/runtime。

## 6. 风险与限制

- 自动关闭任务容易误伤；
- 只按目录判断状态会产生假象；
- 长 comment 链可能造成执行器超时；
- 任务粒度过大时应拆分，但不能无限碎片化。

## 7. 待裁决事项

- review SLA 使用 30 分钟、60 分钟还是按优先级区分；
- done 自动归档是否启用；
- failed 是否允许自动重试。

## 8. 后续动作

- 建立 `任务生命周期状态表`；
- 建立 `review 分诊模板`；
- 建立 `done 归档检查清单`；
- 建立 `blocked 人工决策清单`。
