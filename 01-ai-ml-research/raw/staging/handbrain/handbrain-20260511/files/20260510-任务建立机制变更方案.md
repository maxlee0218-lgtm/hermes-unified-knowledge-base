# 任务建立机制变更方案

> 变更：去掉 GitHub 领任务，改为首脑直接建立
> 状态：提案待确认

---

## 一、当前机制（拟去掉）

```
ChatGPT → GitHub tasks/inbox → PocketClaw → task_inbox_check.sh → Multica Issue → Agent
```

## 二、新机制（提案）

```
用户告知任务 → 首脑直接创建 Multica Issue → Agent 执行
```

## 三、变更内容

### 3.1 去掉的组件

| 组件 | 说明 |
|------|------|
| GitHub tasks/inbox | 不再作为任务入口 |
| task_inbox_check.sh | 不再自动拉取 |
| PocketClaw 定时触发 | 不再触发任务拉取 |
| inbox loop | 停止运行 |

### 3.2 保留的组件

| 组件 | 说明 |
|------|------|
| review loop | 继续巡检 |
| heartbeat | 继续记录 |
| Multica Issue | 继续作为任务载体 |
| Agent 执行 | 继续执行 |

### 3.3 新增的职责

| 职责 | 说明 |
|------|------|
| 首脑直接创建 Issue | 用户告知后，首脑创建 Multica Issue |
| 首脑指派 Agent | 直接指派给研究与架构官 |
| 首脑验收 | 任务完成后验收 |

## 四、执行方式

### 4.1 用户告知任务

用户直接描述任务：
- 任务目标
- 任务类型
- 优先级

### 4.2 首脑创建 Issue

```bash
multica issue create \
  --title "任务标题" \
  --description "任务描述" \
  --assignee "研究与架构官" \
  --priority high \
  --status in_progress
```

### 4.3 Agent 执行

Agent 自动执行，输出报告。

### 4.4 首脑验收

检查报告，转 done。

## 五、优缺点

### 优点

- 简化流程，减少依赖
- 响应更快
- 减少 GitHub 同步问题

### 缺点

- 需要首脑手动创建
- 缺少任务文件记录
- 不利于异步协作

## 六、结论

**提案：去掉 GitHub 机制，改为首脑直接建立。**

等待用户确认。
