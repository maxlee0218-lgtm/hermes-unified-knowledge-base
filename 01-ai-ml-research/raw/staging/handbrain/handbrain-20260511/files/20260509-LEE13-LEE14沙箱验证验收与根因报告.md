## 四、治理原则沉淀

### 4.1 空表/陈旧表排查标准流程

```
发现空表/陈旧表
  → 检查DolphinScheduler调度引用（t_ds_task_definition / t_ds_process_definition）
  → 检查工作流状态（release_state）
  → 检查上游数据是否更新
  → 检查SQL逻辑是否正确
  → 判断：调度问题 / 数据问题 / SQL问题 / 业务废弃
  → 输出：根因分析 + 治理建议（不动生产）
```

### 4.2 生产保护原则
- 发现空表/陈旧表时，先查调度引用和工作流状态
- 不直接改表、不写库、不改生产配置
- 只输出建议，由业务确认后实施

### 4.3 调度优先原则
- 大多数空表/陈旧表问题是调度配置问题，不是数据问题
- 优先检查DolphinScheduler，再检查SQL逻辑，最后检查数据源
