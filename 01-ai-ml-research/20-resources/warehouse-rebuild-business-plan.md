# 数仓重构业务方案（AI智能体驱动 / 生产冻结 / Windows先行验证版）

> 创建时间：2026-05-10  
> 更新原因：明确本次重构的本质目标不是单纯 SQL 改造，而是建设 AI 智能体参与的标准数仓建造、开发、监控、排查与后续读数体系。  
> 状态：业务方案基线  
> 适用范围：数仓重构、Windows测试环境、dbt模型、速程监控、Agent协同  
> 核心原则：生产不动，Windows先行，AI参与开发与治理，AI只建议不碰生产，用户人工决策生产动作

---

## 一、业务背景

当前生产数仓已经在运行，虽然存在口径不统一、链路复杂、历史补丁多、部分表质量不高、调度和同步不够规范等问题，但生产环境已经承载真实业务，预计已有一百到两百张表在被实际使用。

因此，本次重构不是推倒生产重来，也不是直接替换现有生产链路，而是在 Windows 测试环境中建立一套新的、可验证、可比对、可逐步推广的数仓改造体系。

更准确地说，本次重构的核心不是“人工重新写一套数仓”，而是建设一套：

```text
AI 智能体辅助的标准数仓建造体系
```

它要让 AI 智能体参与：

- 重构前摸排；
- 数仓设计；
- SQL / dbt 模型开发；
- 调度与链路分析；
- 数据质量监控；
- 异常排查；
- 新旧链路比对；
- 生产风险发现；
- 后续基于数据的问答和读数。

---

## 二、业务目标

本次数仓重构的业务目标不是“技术上更先进”，而是解决以下问题：

1. 让数仓链路更清楚：知道每张关键表从哪里来、怎么加工、谁在用。
2. 让指标口径更稳定：减少报表层补丁、辅助表口径漂移、临时 SQL 修修补补。
3. 让数据质量更可监控：及时发现空表、陈旧表、断流、调度下线、行数异常。
4. 让新旧链路可比对：新链路在 Windows 跑顺后，再与生产结果对账。
5. 让生产风险最小化：生产继续运行，不由 AI 自动修改任何生产配置或数据。
6. 让后续重构可复制：先做一条样板链路，再逐步推广。
7. 让 AI 智能体成为数仓开发、监控、排查问题的长期助手，而不是一次性聊天工具。
8. 让后续“读数”可以由 AI 基于标准模型、血缘、指标口径和监控结果进行，而不是每次重新翻 SQL。

## 三、总体原则

### 3.1 生产冻结原则

生产环境当前能够运行，必须保持稳定。

禁止 AI 或 Agent 自动执行以下动作：

- 修改生产数据库；
- 修改生产表结构；
- 修改生产 DolphinScheduler；
- 修改生产 DataX；
- 重跑生产调度；
- 补生产数据；
- 删除生产表；
- 自动上线新 SQL；
- 自动恢复生产服务；
- 自动调整白名单、密钥、账号权限。

所有生产优化、上线、修复、补数、调度变更，必须由用户人工确认和人工执行。

### 3.2 Windows 先行验证原则

Windows 是本次新数仓改造的测试环境和验证场。

Windows 上可承载：

- GPT / Agent 执行；
- 测试 Dolphin；
- 测试数仓；
- dbt 模型；
- SQL 沙箱；
- 速程监控；
- 新旧结果比对；
- 目录与资产治理。

新链路必须先在 Windows 测试环境中跑通，再考虑是否进入生产。

### 3.3 AI 参与建设但不碰生产原则

AI / Agent 的职责是：

- 摸排；
- 分析；
- 建议；
- 设计；
- 生成 SQL 草案；
- 生成 dbt 模型草案；
- 沙箱验证；
- 新旧比对；
- 输出报告；
- 标注风险；
- 提醒用户决策；
- 后续基于标准模型和指标口径辅助读数。

AI / Agent 不拥有生产执行权。

## 四、业务范围

### 4.1 第一阶段重点业务范围

第一阶段不做全域重构，只做样板链路。

推荐第一条样板链路：

```text
ADS_SC_XL_13
```

重点关注：

```text
ads_sc_xl_13_defined_manuf_line_name_combined_002
```

原因：

- 用户熟悉；
- 历史问题多；
- 已经有上下游链路基础；
- 适合验证 ODS/DWD/ADS 重构方法；
- 适合测试 dbt 模型、Dolphin 调度、速程监控和新旧比对；
- 适合验证 AI 智能体辅助开发、监控、排查问题的完整工作模式。

### 4.2 第二关注范围

```text
ads_gx_xs_* 数据质量链路
```

该链路适合作为数据质量治理样板，用于验证：

- 空表识别；
- 陈旧表识别；
- release_state 检查；
- 上游是否更新；
- 调度是否存在；
- 速程监控规则是否有效。

## 五、业务分阶段方案

### 阶段 0：全貌摸排

目标：先把当前真实情况摸清楚。

范围：

- 生产数仓现状；
- Windows C/D 盘资产；
- 四台机器 1500 / 111 / 184 / Windows；
- 项目文件；
- 端口进程；
- 隧道转发；
- Dolphin / DataX；
- 速程监控；
- 三类 Agent 角色；
- 历史报告与 GitHub/Wiki 文档。

产出：

- 数仓重构前全量摸排基线报告；
- Windows C/D 盘资产报告；
- 四机运行资产报告；
- 三类 Agent 协同规范；
- 重构前尽调清单。

### 阶段 1：Windows 测试环境治理

目标：把 Windows 测试环境变成可控的新数仓实验场。

重点：

- 明确 C 盘 / D 盘资产；
- 明确哪些项目迁移到 D 盘；
- 统一目录结构；
- 明确 GPT / Dolphin / dbt / 测试数仓 / 速程监控边界；
- 防止测试环境误写生产；
- 确认测试环境能稳定运行。

产出：

- Windows 测试环境运行规范；
- D 盘统一目录方案；
- C 盘迁移候选清单；
- 测试环境隔离边界。

### 阶段 2：最小链路重构验证

目标：只选一条链路做样板。

推荐链路：

```text
ODS镜像
→ DWD标准层
→ ADS_SC_XL_13
→ 速程监控
→ 与生产结果比对
```

技术建议：

- DataX 继续负责同步；
- Dolphin 负责编排；
- dbt 负责模型、血缘、测试、文档；
- 速程监控负责质量护栏；
- Agent 负责分析与报告。

产出：

- dbt 模型草案；
- 新旧结果比对报告；
- 数据质量测试报告；
- 上线风险评估；
- AI 智能体参与数仓开发和排查的标准流程。

### 阶段 3：新旧并跑与人工决策

目标：新链路在 Windows 测试环境中稳定运行一段时间，与生产旧链路进行比对。

要求：

- 行数一致性；
- 指标一致性；
- 时间窗口一致性；
- 异常数据解释清楚；
- 业务口径确认；
- 性能评估；
- 回滚方案。

只有用户确认后，才进入生产改造建议阶段。

### 阶段 4：生产改造建议，不自动执行

目标：形成生产变更建议包。

包括：

- 变更内容；
- 影响范围；
- SQL / 调度 / 同步调整建议；
- 回滚方案；
- 验收标准；
- 风险清单。

生产执行仍由用户人工操作。

### 阶段 5：AI 辅助读数与长期运营

目标：在标准数仓、标准模型、标准指标口径和数据质量监控的基础上，让 AI 能够辅助用户读数、解释异常、定位原因、生成分析结论。

前提：

- 核心指标口径已固化；
- 模型血缘清楚；
- 表和字段有文档；
- 数据质量规则已建立；
- 速程监控能发现异常；
- AI 只能读数和解释，不直接修改生产。

## 六、速程监控定位

速程监控不是普通 BI 大屏，而是生产数据护栏。

它的业务定位是：

```text
监控生产数据不要翻车。
```

重点监控：

- 空表；
- 陈旧表；
- 调度下线；
- 数据断流；
- 行数突变；
- 日期异常；
- 上游未更新；
- DataX 增量异常；
- Dolphin 失败或未触发。

速程监控应优先服务于“生产安全”，而不是做成普通展示大屏。

---

## 七、dbt 业务定位

dbt 当前不替代 Dolphin，也不直接接管生产。

dbt 在本方案中的定位是：

- 模型标准化；
- SQL 分层管理；
- 血缘图；
- 字段文档；
- 数据质量测试；
- 新旧链路比对辅助；
- 为后续 AI 读数提供标准化模型和指标语义基础。

第一阶段只在 Windows 测试环境中使用 dbt。

---

## 八、AI 智能体参与方式

本次重构要验证的不仅是新数仓链路，还包括 AI 智能体参与数仓工作的标准模式。

### 8.1 AI 参与开发

AI 可以辅助：

- 读取旧 SQL；
- 梳理血缘；
- 生成 dbt 模型草案；
- 生成数据质量测试；
- 生成新旧比对 SQL；
- 生成调度建议；
- 生成文档。

### 8.2 AI 参与监控

AI 可以辅助：

- 读取速程监控结果；
- 解释空表、陈旧表、断流、行数波动；
- 识别 Dolphin/DataX 可能原因；
- 输出风险等级；
- 提醒用户是否需要人工处理。

### 8.3 AI 参与排查问题

AI 可以辅助：

- 分析调度状态；
- 分析增量条件；
- 分析上游是否更新；
- 分析 SQL 逻辑；
- 对比新旧链路结果；
- 形成根因报告。

### 8.4 AI 参与读数

AI 后续可以基于标准数仓：

- 查询指标；
- 解释指标变化；
- 识别异常；
- 生成业务分析摘要；
- 给出待确认问题。

但 AI 读数必须基于标准口径、标准模型和数据质量状态，不能基于散乱 SQL 自行发挥。

## 十一、当前业务结论

当前方案判断：

```text
Conditional Go
```

意思是：

- 可以继续推进重构前尽调；
- 可以继续建设 Windows 测试环境；
- 可以准备 dbt 样板链路；
- 可以设计 AI 智能体参与开发、监控、排查、读数的标准流程；
- 不允许动生产；
- 不允许直接进入生产重构。

当前最优主线：

```text
全貌摸排
→ Windows测试环境治理
→ AI智能体工作模式固化
→ ADS_SC_XL_13样板链路
→ dbt模型与速程监控验证
→ 新旧比对
→ 人工决策生产改造
→ AI辅助读数
```

## 十二、禁止事项

在用户明确授权前，禁止：

- 原地改造生产数仓；
- 直接替换生产 ADS；
- 自动上线 Dolphin；
- 自动修改 DataX；
- 自动重跑生产任务；
- 自动修复生产数据；
- 自动迁移 C 盘项目；
- 自动关闭旧链路；
- 自动删除历史表或历史脚本；
- AI 基于未确认口径直接给业务结论。
