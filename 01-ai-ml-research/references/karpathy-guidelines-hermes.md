## 核心原则（4条）

### 1. Think Before Coding
**不要假设。不要隐藏困惑。提出权衡。**

实施前必须：
1. 明确陈述假设。不确定就问。
2. 多种解释？呈现出来——不要默默选择。
3. 有更简单的方法？说出来。必要时推回。
4. 有不清楚的？停下。指出困惑之处。问。

**测试：** 如果你要写代码但还没陈述假设，你就错了。

### 2. Simplicity First
**最小代码解决问题。不做推测性实现。**

规则：
- 不超需求实现功能
- 单用代码不做抽象
- 不添加未要求的"灵活性"
- 不处理不可能场景的错误
- 200行能写成50行，重写

**自问：** "资深工程师会说这过度复杂吗？"如果是，简化。

### 3. Surgical Changes
**只碰必须碰的。只清理自己的mess。**

编辑现有代码时：
- 不"改进"相邻代码、注释或格式
- 不重构没坏的东西
- 匹配现有风格，即使你做法不同
- 发现无关死代码，提一下——不删除

你的改动产生孤儿时：
- 删除**你的**改动导致的未使用导入/变量/函数
- 不删除预先存在的死代码，除非被要求

**测试：** 每行改动都应直接追溯到用户的请求。

### 4. Goal-Driven Execution
**定义成功标准。循环直到验证。**

把命令式任务转化为可验证目标：

| 不要说... | 转化为... |
|-----------|-----------|
| "添加验证" | "为无效输入写测试，然后让它们通过" |
| "修复bug" | "写复现bug的测试，然后让它通过" |
| "重构X" | "确保测试前后都通过" |
| "让它工作" | "定义'工作'的含义，写测试，验证" |

多步骤任务，陈述简要计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

**强成功标准**让你独立循环。
**弱标准**（"让它工作"）需要不断澄清。

## Hermes集成

此skill与其他Hermes skills协同：

- [[test-driven-development]] — 配合原则4（目标驱动）。TDD是验证循环。
- [[systematic-debugging]] — 配合原则1（编码前思考）。先调试，再修复。
- [[writing-plans]] — 配合原则4。多步骤任务需要带验证的计划。
- [[requesting-code-review]] — 四条原则都适用于review。

## 反模式速查

| 原则 | 反模式 | 修复 |
|-----------|-------------|-----|
| Think Before Coding | 默默假设格式、字段、范围 | 列出假设，请求澄清 |
| Simplicity First | 单函数用策略模式 | 一个函数直到需要复杂度 |
| Surgical Changes | 修复bug时重排引号 | 只改修复问题的行 |
| Goal-Driven | "我会review并改进" | "写测试→修复→验证" |

## 关键洞察

Karpathy：

> "LLMs非常擅长循环直到达到特定目标...不要告诉它做什么，给它成功标准，看着它跑。"

好代码简单解决今天的问题，不 premature 解决明天的问题。
